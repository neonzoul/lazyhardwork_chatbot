§ 05

LLM Integration

วิธีที่บอทใช้ Claude และจัดการ Knowledge Base

5.1

KB Injection — Full-Context Strategy

kb.js อ่านไฟล์ knowledge/kb.md ครั้งเดียวตอน server เปิด (Node module cache เก็บไว้ในหน่วยความจำ) ทุกครั้งที่มี LLM call, buildSystemPrompt() ฉีด KB ทั้งก้อนเข้าไปใน system prompt:

system prompt (ทุก call):
  [persona + กฎลงท้าย "[Agent Lay]" ทุก response]
  [ข้อมูลติดต่อจาก config — ไม่ hard-code]
  [KB ทั้งก้อน: identity · บริการ · ราคา · proof · FAQ · handoff · voice samples]
  [กฎ: KB-only · ห้ามแต่ง · escalate unknowns]
  [ถ้านอกเวลา: แนบ notice]

messages: history (≤10 ข้อความ) + joined burst จาก pendingMessages

ทำไมถึงเลือก Full Injection ตอนนี้

KB มีขนาดเล็กโดยออกแบบ (~10 blocks, ~800–1,500 tokens) ค่า Haiku input ($0.25/MTok): ~฿0.01 ต่อ 1,000 calls สำหรับส่วน KB — น้อยมากจนไม่มีนัยสำคัญ ข้อดีของความเรียบง่ายมีมาก: ไม่ต้องมี vector DB, embedding pipeline, ไม่มีปัญหา retrieval miss

เมื่อไหร่ Full Injection จะพัง

ขนาด KBประมาณ Tokensปัญหา
ปัจจุบัน (~5 KB)~1,500ไม่มี — ถูกต้องแล้ว
~20 หน้า~10,000ค่าใช้จ่ายเริ่มโต ยังพอรับได้
~100 หน้า~50,000"Lost in the middle" — คุณภาพตก + ค่าสูงขึ้นชัดเจน
~500 หน้า~250,000เกิน context window ในทางปฏิบัติ — เป็นไปไม่ได้
สัญญาณที่ควรเปลี่ยนไปใช้ RAG
  • → KB เกิน ~30 หน้า (~15K tokens)
  • → KB gap alerts พุ่งขึ้น ทั้งที่ KB มีข้อมูลนั้นอยู่ (retrieval failure)
  • → ค่า input token กลายเป็นสัดส่วนสำคัญของต้นทุนต่อ conversation

RAG Architecture (อ้างอิงสำหรับอนาคต)

Startup:
  chunk KB → sections ~300 tokens
  embed แต่ละ chunk (Anthropic Embeddings / text-embedding-3-small)
  เก็บใน vector index (in-memory / Chroma / Pinecone)

Per message:
  1. Embed คำถามของผู้ใช้
  2. Cosine-similarity search → top-3 chunks (~1,500 tokens)
  3. ฉีดเฉพาะ chunks ที่เกี่ยวข้องใน system prompt
  4. LLM ตอบจากตัวอย่างที่โฟกัส ไม่ใช่ทั้งห้องสมุด

Trade-off: ต้นทุนต่ำ + คุณภาพดีขึ้นใน KB ขนาดใหญ่
           vs ความซับซ้อนของ infrastructure ที่เพิ่มขึ้น
5.2

Multi-Message Burst — Lossless Accumulator

ผู้ใช้ LINE มักส่งหลายข้อความต่อกันเป็นชุด debounce window 12 วิ เก็บทุกข้อความ ไว้ใน session.pendingMessages[] เมื่อ timer ยิง จะรวมทุกข้อความด้วย newline แล้วประมวลผลเป็น block เดียว — LLM เห็น intent ครบทั้งหมด ไม่ใช่แค่ข้อความสุดท้าย

User: "สวัสดี" → push pendingMessages, reset 12s timer
User: "ผมสนใจบอท" → push pendingMessages, reset 12s timer
User: "ราคาเท่าไหร่" → push pendingMessages, reset 12s timer
12s ไม่มีข้อความใหม่ → drain: "สวัสดี\nผมสนใจบอท\nราคาเท่าไหร่"
→ processMessage(joined burst)

ค่า 12 วิ ปรับได้ผ่าน DEBOUNCE_MS ใน bot.js — เพิ่มถ้าผู้ใช้พิมพ์ช้า, ลดถ้า latency สำคัญกว่า

5.3

KB Gap & Lead Detection

  • →
    KB Gap (heuristic): ถ้า reply มีคำว่า ไม่มีข้อมูล หรือ ไม่ทราบ → ส่งคำถามของผู้ใช้ไป Discord #crm เพื่อให้ Mos อัปเดต KB ทีหลัง
  • →
    Lead Detection: ปัจจุบัน manual trigger (ผู้ใช้พูด buying-signal phrases) Phase 2: structured extraction ผ่าน LLM call เพิ่มเติมเมื่อ handoff ยิง

7.2 — agents.config.js (Model Definitions)

กำหนด Model ต่อ Agent Role — version-controlled

// src/agents.config.js
// แก้ที่นี่เพื่อเปลี่ยน model — ไม่ต้องยุ่งกับ .env
module.exports = {
  agentLay: {
    model:       'claude-haiku-4-5-20251001',  // เร็ว ถูก เหมาะภาษาไทย KB-only
    maxTokens:   1024,
    description: 'LINE OA reply bot — KB-only answers',
  },
  // เพิ่ม sub-agent ในอนาคตที่นี่:
  // crmSummarizer: { model: 'claude-haiku-4-5-20251001', maxTokens: 512 }
  // reportAgent:   { model: 'claude-sonnet-4-6',         maxTokens: 4096 }
};
เหตุผลที่แยก
Secrets (.env) = ไม่เคย commit — API keys, tokens, webhook URLs
Model selection (agents.config.js) = ไม่ใช่ความลับ — เป็น engineering decision ที่ควร version-control, review ใน PR ได้, เปลี่ยนโดยไม่ต้อง redeploy การเปลี่ยน model ของ agent หนึ่งต้องไม่กระทบ agent อื่น

05
Wire LLM (llm.js + kb.js) สำหรับ BOT_ACTIVEทดสอบ KB-only answers — ถาม out-of-KB แล้วต้องไม่ hallucinate
06
Handoff trigger → WAITING_HUMAN + discord.js #crm notifyพิมพ์ "คุยกับคนจริง" → state เปลี่ยน → Discord แจ้ง
07
Lead detection heuristic + #leads Discord notifyผู้ใช้แสดง buying signal → Discord #leads ได้รับ payload
08
ทดสอบ out-of-hours behaviorMock ชั่วโมง หรือเปลี่ยน BUSINESS_HOURS_START ชั่วคราว
09
